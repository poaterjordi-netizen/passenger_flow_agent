## 泛化回答机制

本轮没有继续为每一种自然语言表述增加一个互斥任务类型，而是采用四层处理顺序：

1. 明确的客流查询、排行、对比、预测、诊断等问题，编译为受控业务 Operation，执行固定或参数化工具；
2. “你能做什么”等问题，读取当前能力注册表并确定性回答，不调用大模型；
3. 有明确语义但不需要客流数据库的问题，进入 `general_answer`，只向 GPT 提供问题和边界证据；
4. 只有缺少分析对象、起终点或其他执行必需信息时，才请求用户补充。

这使以下问题获得稳定处理：

- “列出数据库中的所有地铁站”进入完整实体发现，不再误报歧义；
- “从北京交通大学到北京工业大学怎么走”进入出行规划，不再追问客流指标；
- “什么是地铁断面客流”进入一次 GPT 通用知识回答，并明确没有查询 `metroflow` 业务行；
- “预测明天北京天气”不会误用客流预测，也不会伪造实时天气，而是指出缺少实时天气数据源。

通用兜底并不等于无条件回答。需要实时天气、新闻、价格、法规或实时运营状态时，系统必须连接相应
外部数据源后才能给出事实性结论；模型记忆不能冒充实时数据。

## 真实数据库与 GPT 链路

本地影子模式已经把 Web、FastAPI、只读 MySQL、证据封装、GPT-5.6 Sol、核验器和运行轨迹连接起来。
启动脚本从仓库外配置和 macOS 钥匙串读取连接信息，浏览器端不接触数据库凭据或 API 令牌。

真实数据仍遵循以下约束：

- 只允许固定、参数化的 QueryIR 查询，不执行模型生成 SQL；
- 默认验证 TLS 服务器身份，并可绑定叶证书 SHA-256 指纹；
- 每次数据库操作开启只读事务，完成后回滚；
- 缺列、`NULL`、字符串、布尔值、非有限数和负客流值均不会被静默转换为零；
- 真实 Evidence 默认不发送给模型，只有端点、模型和策略精确匹配时才允许最小化出域；
- 数据库凭据、物理表清单、真实行、查询结果、审计和预测产物均不进入 Git。

## 前后端可见结果

后端新增能力发现和治理状态接口，前端直接使用同一组契约显示当前运行范围。用户现在可以看到：

- 本次问题编译出的 Operation 和匹配能力；
- 使用了确定性回答、证据合成还是通用 GPT 回答；
- 工具是否查询真实数据库、结果是否完整、是否发生截断；
- 数据城市、角色、时间范围、粒度、来源版本和质量状态；
- 模型实际调用次数、模型名称、出域是否批准以及调用级审计；
- 当前生产准入门为什么仍处于阻断状态。

这让治理规则不再只是仓库中的配置文件，而是会影响后端能否创建会话、注册哪些工具，以及前端按钮和状态提示。

## 仍未完成的生产条件

当前 `production-readonly` promotion gate 保持 `blocked_pending_approval`，这是有意的真实状态。正式上线仍需要：

- 业务负责人、数据负责人、安全负责人和工程负责人的明确签字；
- 权威站点、线路、活动、公交、GIS、实时运营和 SOP 数据源；
- 真实业务 Gold Cases、预测准确率、超时、并发、成本和数据库负载验收；
- 正式身份提供方、多人权限隔离、加密轨迹存储和审计保留策略；
- 经过批准的模型端点、数据出域规则和人工运营责任流程。

因此，本轮成果应表述为“能够连接真实只读数据和真实 GPT 的本地受治理影子原型”，不能表述为已经上线的生产客运决策系统。

## 后续建议

下一阶段不应继续扩大 Prompt 中的固定任务枚举，而应优先为现有 Operation/能力层接入权威工具：

1. 接入实时地铁、公交、天气、活动和地图 API，使通用问题能够转为有来源的工具回答；
2. 建立权威站点与线路主数据，区分“观测到的实体”与“全量主数据”；
3. 由业务人员批准真实 Gold Cases 和指标口径，持续评估 GPT 相对确定性基线的净增益；
4. 完成正式身份、权限、负载和出域验收后，再逐项解除 promotion gate。
